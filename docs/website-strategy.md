# 행사어때 웹사이트 및 플랫폼 구축 초안

## 1. 목표

행사어때 웹사이트는 마이스파트너가 어떤 서비스를 제공하는 회사인지 보여주는 기업 홈페이지여야 한다.

초기 목표는 다음 두 가지다.

- 마이스파트너의 전문성과 운영 역량을 분명하게 전달한다.
- 서비스 소개, 포트폴리오, 문의 전환이 자연스럽게 이어지는 구조를 만든다.

## 2. 주요 사용자군

- 공공기관 및 지자체 담당자
- 대학, 학회, 연구기관 행사 담당자
- 기업 마케팅/세일즈/브랜드 행사 담당자
- 행사 운영사 및 협력 파트너

## 3. 웹사이트 핵심 콘텐츠

- 히어로 메시지: 대전형 MICE 행사 통합 운영 플랫폼
- 서비스 설명: 행사 기획, 운영, 홈페이지 제작, 참가자 응대, 현장 체크인 지원
- 활용 시나리오: 포럼, 학회, 기업 행사, 설명회, 현장 설치사례
- 운영 프로세스: 기획, 구축, 현장 운영, 사후 보고
- 신뢰 요소: 운영 방식, 파트너십, 실적, 문의 전환

## 4. 프론트엔드 권장 구조

- `src/App.tsx`
  - 현재는 브랜드 랜딩을 보여주는 메인 엔트리
- 향후 권장 분리
  - `src/pages/HomePage.tsx`
  - `src/pages/ServicesPage.tsx`
  - `src/pages/CaseStudiesPage.tsx`
  - `src/pages/ContactPage.tsx`
  - `src/components/sections/*`
  - `src/components/ui/*`
  - `src/content/*`

권장 기술 방향:

- React + TypeScript + Vite 유지
- Tailwind 기반 디자인 토큰 유지
- 라우팅 필요 시 `react-router-dom` 도입
- 정적 콘텐츠는 초기에 로컬 데이터 파일로 관리
- 문의 폼과 데모 요청 폼은 API 연동 전에도 구조를 먼저 컴포넌트화

## 5. 백엔드 권장 구조

초기에는 문의 저장과 관리자 확인을 위한 API만 먼저 두고, 이후 회사 홈페이지 운영에 필요한 기능을 순차 확장한다.

핵심 도메인:

- `organizations`
- `inquiries`
- `portfolio_entries`
- `notices`

권장 스택:

- API: Express 또는 NestJS
- DB: PostgreSQL
- 인증: 역할 기반 로그인
- 파일: S3 호환 스토리지
- 외부 연동: 이메일, 문자/알림톡, 결제

## 6. 단계별 로드맵

### Phase 1. 브랜드 사이트

- 메인 랜딩
- 서비스 소개
- 활용 사례
- 문의 전환

### Phase 2. 홈페이지 운영 고도화

- 포트폴리오 상세 관리
- 공지/자료실 관리
- 문의 회신 상태 관리
- 관리자 로그인

### Phase 3. 운영 고도화

- 세션 관리
- 운영 인력 권한 분리
- 실시간 운영 대시보드
- 결과 리포트 자동화

## 7. 현재 작업 상태

- 메인/서비스/포트폴리오/자료실/회사소개/문의 페이지로 구조 분리 완료
- 포트폴리오 상세 템플릿 라우트 추가 완료
- 문의 저장 API 및 로컬 SQLite 저장 구조 추가
- 운영관리 초안 페이지에서 문의 목록/상태 확인 가능
- 회사 실데이터와 법적 문서는 placeholder 상태로 유지
- 이후 우선 작업 추천 순서:
  1. 관리자 로그인 및 권한
  2. 문의 회신 이력/메모 고도화
  3. 자료실/공지 관리 구조 추가
